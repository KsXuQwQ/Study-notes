## 关于Nginx反向代理

Nginx：将前端发送的动态请求由 Nginx转发到后端

Nginx反向代理的好处：

- 提高访问速度  （客户端再次请求相同内容时，Nginx可以直接从自己的缓存中返回数据，从而避免重复向后端服务器发起请求，达到加速访问和减轻后端压力的效果）
- 负载均衡： 将大量的请求按照指定方式均衡分配给集群中的每台服务器
- 保证后端服务器安全：事实上前端不能直接访问后端（不安全），而是需要先发给nginx走内网访问后端服务
### Nginx反向代理的配置方式

```
server{
       listen 80;
       server_name localhost;
       
       location /api/{ 
                      proxy_pass http://localhost:8080/admin/ #反向代理 （admin后有 / 意味着匹配Api替换后，在其后加上替换前的请求）
       }

}
```
- **`listen 80;`**：Nginx 监听本机的 80 端口（HTTP 默认端口）。
- **`server_name localhost;`**：只有当你通过 `localhost` 这个域名（或 127.0.0.1）访问时，这个配置才会生效。多个相同端口号的情况下，指定进入哪个配置时，通过此命令区分。
- location /api/：当请求匹配到 `/api/` 路径时，触发转发
1. `location` 匹配到的是**前缀**（如 /api/`）
    
2. Nginx 会从原始 URI 中**去掉**这个前缀
    
3. 然后把**剩余部分**拼接到 `proxy_pass` 的路径后面
### Nginx负载均衡配置

```
upstream webservers（webservers这个就是服务器需要均衡访问的地址）{
    server 192.168.100.128:8080;
    server 192.168.100.129:8080;
}

server{
    listen 80;
    server_name localhost;

    location /api/ {
    proxy_pass http://webservers（与upsatream对应，不限制名字只要对应）/admin; #负载均衡
    }
}
```
### 关于负载策略
```
upstream backend {
    # 方式1：轮询（默认）
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式2：权重
    server 192.168.1.10:8080 weight=3;
    server 192.168.1.11:8080 weight=1;

    # 方式3：IP Hash
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式4：最少连接
    least_conn;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式5：URL Hash（需第三方模块）
    hash $request_uri consistent;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式6：fair（需第三方模块）
    fair;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```
轮询：	默认方式。每个请求按时间顺序逐一分配到不同的后端服务器，如果后端服务器宕机，能自动剔除。

weight	：权重方式。默认为1，权重越高，被分配的客户端请求就越多。用于后端服务器性能不均的情况。

ip_hash：依据 IP 分配方式。每个请求按访问 IP 的 hash 结果分配，这样每个访客可以固定访问一个后端服务，解决 Session 共享问题。

least_conn：依据最少连接方式。把请求优先分配给连接数最少的后端服务，适用于请求处理时间长短不一的情况。

url_hash：依据 URL 分配方式。按访问 URL 的 hash 结果来分配请求，使每个 URL 定向到同一个后端服务器，适用于缓存场景。

fair：依据响应时间方式。响应时间短的服务将会被优先分配，需要安装第三方模块（nginx-upstream-fair）。

## 完善登录功能MD5

一般的 数据库存储用户密码都是明文存储，这在企业中很危险，所以我们使用MD5加密方式对密码加密。

使用Spring提供的 DigestUtils 工具类 

```
 password = DigestUtils.md5DigestAsHex(employeeLoginDTO.getPassword().getBytes());
```
这样前端输入密码会被加密，而加密后的密码将会比对，这样前端密码在不变的情况下完成了加密。

## 统一返回结果 


```
/**  
 * 后端统一返回结果  
 * @param <T>  
 */  
@Data  
public class Result<T> implements Serializable {  
  
    private Integer code; //编码：1成功，0和其它数字为失败  
    private String msg; //错误信息  
    private T data; //数据  
  
    public static <T> Result<T> success() {  成功，但不返回数据
        Result<T> result = new Result<T>();  
        result.code = 1;  
        return result;  
    }  
  
    public static <T> Result<T> success(T object) {  ### 成功，返回数据
        Result<T> result = new Result<T>();  
        result.data = object;  
        result.code = 1;  
        return result;  
    }  
  
    public static <T> Result<T> error(String msg) {  ### 失败，返回错误信息
        Result result = new Result();  
        result.msg = msg;  
        result.code = 0;  
        return result;  
    }  
  
}
```
**统一响应封装类**
把所有接口的返回结果**统一成一种格式**，方便前端统一处理
这个项目采用了Result类统一返回，这样可以代码可读性更好，且更优雅

这里需要理解 ”成功，返回数据“的（public static <T> Result<T> success(T object)）流程

假如返回 user变量
这时方法/形参中的 T会找到user变量的类 从而使object变为User类，而方法返回值Result<T>表示Result装了User类型的数据，
Result<T> result = new Result<T>() 创建的是 Result 类型的对象，带 User 泛型
这样带了User泛型的 result，可以被带了User泛型object赋值，从而返回给前端

## JWT令牌认证流程

1. 前端用户认证 提交用户名 和 密码 后端认证通过 
2. 后端生成JWT Token返回给前端 前端存储JWT Token
3. 前端发起请求后端接口 每次发送请求头中 都会携带JWT Token 后端进行拦截请求并验证Token
4. 后端验证Token分为 验证通过或不通过
5. 通过 返回给前端展示请求接口的数据（在执行业务逻辑之后），不通过返回错误信息，并让前端回到登录界面

时间线：

登录时：
  后端：从数据库查出 1001 ID → 用代码生成 JWT 字符串 → 返回给前端
  前端：收到 JWT 字符串 → 存到 localStorage

后续请求时：
  前端：从 localStorage 取出 JWT → 塞到请求头里 → 发送请求
  后端：从请求头取出 JWT → 解析出 1001 → 存到 ThreadLocal

### JWT令牌认证流程详解
为了更清晰的理解JWT令牌的流程，现按步骤仔细说明

#### 登录通过给予令牌
```
// 1. 创建一个 Map，存放要放进 JWT 的数据
Map<String, Object> claims = new HashMap<>();
claims.put(JwtClaimsConstant.EMP_ID, employee.getId());  // 存放员工ID

// 2. 调用 JwtUtil 生成 JWT 令牌
String token = JwtUtil.createJWT(
    jwtProperties.getAdminSecretKey(),  // 密钥（配置文件读取）
    jwtProperties.getAdminTtl(),        // 过期时间（配置文件读取）
    claims                              // 要存放的数据
);

// 3. 构建返回给前端的对象
EmployeeLoginVO employeeLoginVO = EmployeeLoginVO.builder()
    .id(employee.getId())          // 员工ID
    .userName(employee.getUsername()) // 用户名
    .name(employee.getName())      // 姓名
    .token(token)                  // JWT令牌
    .build();
```
**代码说明：**

- `claims` 是一个 Map，存放要放进 JWT 的数据（键值对）
    
- `JwtUtil.createJWT()` 把 Map 变成加密字符串
    
- `EmployeeLoginVO` 是返回给前端的对象（不包含密码等敏感信息）
#### 拦截器（Interceptor）
首先我们需要拦截器，拦截器是由Spring框架提供的组件，它会在后端接受前端的请求（Colltroller层）之前运行，我们能在进入核心请求之前完成两件事：
1. 身份认证
2. 数据注入

#### 使用拦截器
首先我们需要在类中实现HandlerInterceptor接口，从而重写preHandle方法 来获得拦截器

```
public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {}
```
首先我们需要清晰实参这三个类是什么：

 HttpServletRequest：它是 Java EE（现在叫 Jakarta EE）规范中定义的接口，由 Servlet 容器（比如 Tomcat）在收到请求时创建。它封装了前端发来的所有请求信息，它可以
 - **获取请求头**：比如你需要的 JWT 令牌，就在这里取：`request.getHeader("Authorization")`。
    
- **获取请求参数**：比如 URL 里带的 `?id=1`，或者表单提交的数据。
    
- **获取请求方法**：是 GET、POST 还是 PUT。
    
- **获取客户端信息**：比如客户端的 IP 地址。

HttpServletResponse：它同样是 Java EE 规范定义的接口，用于构建将要返回给前端的响应。它封装了后端想发给前端的所有信息。
它可以
- **设置响应状态码**：比如校验失败时，你可以设置 `response.setStatus(HttpServletResponse.SC_UNAUTHORIZED)` 返回 401 状态码。
    
- **设置响应头**：比如设置 `Content-Type`。
    
- **写入响应内容**：你可以直接向 response 的输出流里写数据返回给前端（虽然通常由 Controller 做，但拦截器也能做）。

Object handler： 它是 Spring 框架**传入的，代表真正要处理这个请求的 Controller 方法**的元数据对象。它可以
- **获取目标方法信息**：你可以通过它知道你这次请求最终要调用的是哪个类、哪个方法。
    
- **实现细粒度权限**：比如你可以给不同方法配置不同权限，在拦截器里解析出 `handler` 对应的方法，判断该方法是否需要某种特定权限。

在知道拦截器的方法后，我们使用三个参数的引用，使用拦截器方法
我们需要让拦截拦截国定的请求，因为前端会向后端发送很多请求，我们只需让前端访问Controller层的时候被拦截即可
```
if (!(handler instanceof HandlerMethod)) {  
    //当前拦截到的不是动态方法，直接放行  
    return true;  因为方法是布尔值
}
```
这里HandlerMethod是Spring 框架中一个非常核心的类，它代表一个 即将被调用的 Controller 方法"的完整描述对象。所以该代码可以用于判读是否为Controller层请求。
接着：
```
String token = request.getHeader(jwtProperties.getAdminTokenName()); 
```
这段代码即可在请求体中获取令牌

```
Claims claims = JwtUtil.parseJWT(jwtProperties.getAdminSecretKey(), token);
```
jwtProperties.getAdminSecretKey()：从配置里取密钥（比如 `"itcast"`），用来验证令牌的签名是否有效

`Claims` 本质上就是一个 Map<String, Object>的“升级版”，专门用来存放 JWT 解析后的数据。
这段代码的作用是：**从解析后的 JWT 令牌数据（`Claims`）中，取出员工 ID，并转换成 `Long` 类型**。
