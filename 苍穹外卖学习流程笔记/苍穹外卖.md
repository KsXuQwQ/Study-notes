整体项目结构

| sky-take-out | maven父工程，统一管理依赖版本，聚合其他子模块                  |
| ------------ | ------------------------------------------ |
| sky-common   | 子模块，存放公共类，例如：工具类，常量类，异常类等                  |
| sky-pojo     | 子模块，存放实体类、==VO、DTO==等                      |
| sky-server   | 子模块，后端服务，存放配置文件、Controller、Service、Mapper等 |
|              |                                            |
==VO：视图对象，为前端展示数据提供的对象==，前端的表单和数据需要后端提供，后端提供的数据封装给到前端的这个对象即为VO

==DTO：数据传输对象，通常用于程序中各层之间转递数据==，比如前端传入数据，那么后端需要接收数据并且封装Java对象，这个时候可以使用DTO

POJO：（从语法上来看，VO，DTO都是POJO，项目细化做了区分）普通Java对象，只有属性和对应的getter和setter
Entity：实体，通常与数据库的表对应

## 前后端联调

Controller层 --> Service层 --> Mapper层

Controller层
1. 接受并封装参数
2. 调用service层方法查询数据库

1. 调用mapper查询数据库

Mapper层

## 关于Nginx反向代理

Nginx：将前端发送的动态请求由 Nginx转发到后端

Nginx反向代理的好处：

- 提高访问速度  （客户端再次请求相同内容时，Nginx可以直接从自己的缓存中返回数据，从而避免重复向后端服务器发起请求，达到加速访问和减轻后端压力的效果）
- 负载均衡： 将大量的请求按照指定方式均衡分配给集群中的每台服务器
- 保证后端服务器安全：事实上前端不能直接访问后端（不安全），而是需要先发给nginx走内网访问后端服务
### Nginx反向代理的配置方式

```
server{
       listen 80;
       server_name localhost;
       
       location /api/{ 
                      proxy_pass http://localhost:8080/admin/ #反向代理 （admin后有 / 意味着匹配Api替换后，在其后加上替换前的请求）
       }

}
```
- **`listen 80;`**：Nginx 监听本机的 80 端口（HTTP 默认端口）。
- **`server_name localhost;`**：只有当你通过 `localhost` 这个域名（或 127.0.0.1）访问时，这个配置才会生效。多个相同端口号的情况下，指定进入哪个配置时，通过此命令区分。
- location /api/：当请求匹配到 `/api/` 路径时，触发转发
1. `location` 匹配到的是**前缀**（如 /api/`）
    
2. Nginx 会从原始 URI 中**去掉**这个前缀
    
3. 然后把**剩余部分**拼接到 `proxy_pass` 的路径后面
### Nginx负载均衡配置

```
upstream webservers（webservers这个就是服务器需要均衡访问的地址）{
    server 192.168.100.128:8080;
    server 192.168.100.129:8080;
}

server{
    listen 80;
    server_name localhost;

    location /api/ {
    proxy_pass http://webservers（与upsatream对应，不限制名字只要对应）/admin; #负载均衡
    }
}
```
### 关于负载策略
```
upstream backend {
    # 方式1：轮询（默认）
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式2：权重
    server 192.168.1.10:8080 weight=3;
    server 192.168.1.11:8080 weight=1;

    # 方式3：IP Hash
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式4：最少连接
    least_conn;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式5：URL Hash（需第三方模块）
    hash $request_uri consistent;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    # 方式6：fair（需第三方模块）
    fair;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```
轮询：	默认方式。每个请求按时间顺序逐一分配到不同的后端服务器，如果后端服务器宕机，能自动剔除。

weight	：权重方式。默认为1，权重越高，被分配的客户端请求就越多。用于后端服务器性能不均的情况。

ip_hash：依据 IP 分配方式。每个请求按访问 IP 的 hash 结果分配，这样每个访客可以固定访问一个后端服务，解决 Session 共享问题。

least_conn：依据最少连接方式。把请求优先分配给连接数最少的后端服务，适用于请求处理时间长短不一的情况。

url_hash：依据 URL 分配方式。按访问 URL 的 hash 结果来分配请求，使每个 URL 定向到同一个后端服务器，适用于缓存场景。

fair：依据响应时间方式。响应时间短的服务将会被优先分配，需要安装第三方模块（nginx-upstream-fair）。

## 完善登录功能MD5

一般的 数据库存储用户密码都是明文存储，这在企业中很危险，所以我们使用MD5加密方式对密码加密。

使用Spring提供的 DigestUtils 工具类 

```
 password = DigestUtils.md5DigestAsHex(employeeLoginDTO.getPassword().getBytes());
```
这样前端输入密码会被加密，而加密后的密码将会比对，这样前端密码在不变的情况下完成了加密。

```
BeanUtils.copyProperties(employeeDTO,employee); 前为拷贝值，后为粘贴值
```

## 统一返回结果 


```
/**  
 * 后端统一返回结果  
 * @param <T>  
 */  
@Data  
public class Result<T> implements Serializable {  
  
    private Integer code; //编码：1成功，0和其它数字为失败  
    private String msg; //错误信息  
    private T data; //数据  
  
    public static <T> Result<T> success() {  成功，但不返回数据
        Result<T> result = new Result<T>();  
        result.code = 1;  
        return result;  
    }  
  
    public static <T> Result<T> success(T object) {  ### 成功，但返回数据
        Result<T> result = new Result<T>();  
        result.data = object;  
        result.code = 1;  
        return result;  
    }  
  
    public static <T> Result<T> error(String msg) {  ### 失败，返回错误信息
        Result result = new Result();  
        result.msg = msg;  
        result.code = 0;  
        return result;  
    }  
  
}
```
**统一响应封装类**
把所有接口的返回结果**统一成一种格式**，方便前端统一处理
这个项目采用了Result类统一返回，这样可以代码可读性更好，且更优雅
